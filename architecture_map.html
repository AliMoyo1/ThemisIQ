Architectural Issues & Notes
Identified patterns, inconsistencies, and improvement opportunities.
Template architecture inconsistency
ARIA uses multi-page Jinja2 extending base_shell.html. Sentinel extends base_shell via its own base.html. GRID and BCM are standalone SPAs including _icon_sidebar.html directly. Launcher pages extend their own standalone base.html. This means 4 different template inheritance strategies coexist.
Launcher routes.py is a mega-file (~3,150 LOC)
Contains auth, admin, workflows, SLA, reports, calendar, analytics, tasks, API keys, webhooks, reminders, trainer, and bulk operations in a single file. Should be split into sub-routers (e.g., admin_router, workflow_router, etc.).
SQLite concurrency limitations
WAL mode helps, but SQLite is single-writer. With 270+ endpoints and async routes, write contention can cause "database is locked" errors under moderate load. Enterprise deployment should consider PostgreSQL migration path.
Duplicate framework tables
GRID has its own grid_frameworks table separate from aria_frameworks. These should be unified — the user's request for cross-module framework sharing confirms this is a gap.
No cross-module document/evidence linking
While evidence_items and evidence_links tables exist, modules don't actively query each other's artifacts. ARIA docs can't be pulled as GRID evidence, BCM plans can't reference ARIA policies. The event bus exists but handlers are minimal.
AI service inconsistency
ARIA uses raw httpx to Anthropic API. GRID/BCM use the anthropic SDK. Sentinel supports 3 providers via httpx. Should standardize on one approach with a shared AI client abstraction.
Platform pages use dark standalone theme
Launcher's base.html has its own dark styling that doesn't use base_shell.html CSS variables. Workflows, reports, calendar, analytics, task board all have a mismatched aesthetic compared to ARIA/Sentinel which use the unified shell.
No automated testing
No test files found in the codebase. For an enterprise compliance tool, this is a significant gap — especially for RBAC, data integrity, and cross-module operations.
